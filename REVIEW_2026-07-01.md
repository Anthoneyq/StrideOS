## ✅ FIXED THIS PASS (verified)

### Coach-flagged correctness
1. **"6:60" pace bug — fully killed (P1).** Both coaches saw a pace render as "6:60" instead of "7:00". A carry-safe helper existed but the *primary* displays bypassed it. Fixed **9 formatter sites**: Coach Split Table per-mile, free-tier hero, builder hero, interval-ladder, performance-curve Y-axis, Strava /km, plus `fmtRepTime`/`fmtSplit`/`fmtT` sub-minute carry. Verified: 14:30 5K @ threshold now shows `5:00`, 16:06 @ easy `7:00`, etc.
2. **Forecast table was internally incoherent (P1).** Each distance row picked its own anchor PR, so adjacent rows could jump (a stale-vs-fresh PR tiebreak flipped anchors → implausible pace steps — the "half vs marathon don't match" family). Now the whole table forecasts from **one pinned anchor** (the athlete's primary PR) so the curve follows a single fatigue model; the athlete's other PRs still personalize it via the fatigue-exponent blend. Verified: realistic athletes now show a smooth monotonic curve (max adjacent step ~6% vs ~22% before).
3. **Sprint predictions hidden for distance runners (P2).** A 5K/1600 runner no longer sees a "100m: 0:14" forecast. Non-observed forecasts ≥3 event-domains away, and all sprint rows (≤400m) for a ≥1500m primary, are now dropped (not just dimmed). Real PRs always show. 800m/sprint primaries still get their full range.
4. **Short-race anchor can't produce impossible paces (P2).** A 400m PR no longer prescribes a "mile at 118%" (that produced sub-world-record miles) or reps beyond ~the race distance. Per-mile is now hidden and reps are capped at ~1.25× the anchor for sub-1500m PRs. (Aerobic zones were already blocked; this closes the race/supra-race hole Doug's "VO2 breaks the world record" pointed at.)
5. **Zone labels no longer read as %VO2max (P3).** The split-table % is now explicitly **"% of race pace"** (header + each row), and the false "uses a VDOT-equivalent estimate" explanation is corrected to describe the actual Canova race-pace model. (The VDOT reconciliation subsystem was dead code — a units bug made it return null for every athlete — so the app was silently advertising a calculation it never ran.)

### Data integrity (cloud sync)
6. **Phantom PRs from a changed primary event (P2).** Editing an athlete's primary event (e.g. 1600m→5K) left the old race row behind, which reappeared as a fake "additional PR" after a cloud reload and silently changed predictions. Now classified by provenance (`source_ref`), not by name — the stale primary is ignored.
7. **Cleared PRs resurrecting after reload (P2).** Removing an additional PR didn't delete its cloud row, so it came back on the next load. `saveAthlete` now reconciles: any local-import race not in the athlete's current PR set is deleted server-side. (Uses the coach's existing RLS; no migration needed.)

### Backend / billing
8. **Strava "Connect" would 401 for every athlete (P1).** `strava-oauth-callback` must run without a JWT (Strava's redirect carries none), but nothing enforced that — the documented deploy commands would turn JWT verification on and silently break every Strava connect. Added declarative `verify_jwt=false` for `strava-oauth-callback` and `stripe-webhook` (and explicit `true` for the three authenticated functions) in `config.toml`, so deploys are correct regardless of CLI flags.
9. **Annual price mismatch $199 vs $144 (P2).** UI/config/strategy say **$199/yr**, but the deploy guard forced $144 and the checkout comment said $144 — you'd have shown one price and charged another. Aligned the deploy validator (`19900¢`), the override phrase, and the edge-function comment to the locked **$199**. `Stripe_Setup_SOP.md` (stale $20/mo Payment-Link doc) got a **SUPERSEDED** banner pointing at the real Edge-Function checkout.

**Verification:** inline JS parses clean, `node prediction_benchmarks.js` = **31/31**, `bash -n deploy-stripe-functions.sh` OK. Diff is surgical (5 files, ~120 lines).

## 🟡 Remaining improvements (lower priority — I can build these on request)
- **Strava OAuth CSRF hardening (P3 security):** the callback trusts the athlete UUID as OAuth `state` with no single-use nonce. Low exploitability (UUIDs act as secrets) but it's an anti-pattern; wants a server-side nonce table.
- **Per-PR dates for additional PRs (P2→now mostly moot):** additional PRs carry no date, so they read as "unknown" freshness. The single-anchor forecast fix (#2) removes most of the impact; the remaining fix (a date input per PR + storing `race_date` on additional-PR rows) is a small feature.
- **Bulk CSV/XLSX import is single-PR only (P3):** imported rosters can't use Multi-Event Analysis until each athlete is re-edited by hand. Add repeatable event:time columns to the import mapping.
- **3200m forecast runs ~1% fast** for a mile-only anchor (10:53 vs coaches' 11:00–11:12) — driven by the flat Riegel/Vickers weight at that ratio, not the Purdy band. Minor; a real fix wants held-out validation (roadmap C1).
- **Near-duplicate forecast rows** (1500m/Mile/1600m and 3000m/3200m/2 Mile all shown) clutter the table with almost-identical times. Optional consolidation.
- **Forecast coherence caveat:** for an athlete with genuinely contradictory PRs (e.g. a 4:20 mile + 17:30 5K), observed PRs will still visibly diverge from the single-anchor forecast — that's correct (it's the event-fit "you're underperforming here" signal), not a bug.
